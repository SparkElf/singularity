# 奇点 L3.1 生产实时协作架构方案

## 1. 目标与前置

本方案承接 [L3.1 PRD](../product/l3.1-realtime-collaboration.md)、[L3.0 语义原型](l3-realtime-collaboration.md) 和 [ADR-031](../adr/0031-l3-production-collaboration.md)。目标是把已验证的语义操作接入真实 React/Protyle、NestJS Gateway 和 Go Kernel，并保持单一正文事实源。

本地证据足够完成架构决策，本轮不进行外部网页搜索。既有 `KernelWebSocketGateway`、`SpaceConnectionRegistry`、`DocumentAccessPolicy`、PostgreSQL 事务、React Query/shadcn/Tailwind 4 和 L3.0 semantic core 已提供边界先例。

## 2. 架构决策摘要

1. 专用双向协作 WebSocket 与既有只接收推送的 Protyle WebSocket 分离；浏览器不能向 Protyle 订阅发送命令。
2. Go Kernel 继续拥有正文、块、AV、历史和 canonical operation log；PostgreSQL 不保存第二份正文或完整正文快照。
3. NestJS 负责认证会话、ACL 重验、功能开关、协作会话生命周期、协议解析、审计和观测；不复制 Go reducer。
4. 首期只支持单 API 副本和单空间 Kernel 协作通道。连接注册表、presence 和未确认操作位于 API 进程内；进程重启后由 Kernel 历史恢复，presence 丢弃。
5. 加密内容库只有在显式“受限协作”开关、文档授权和 Kernel 内容密钥同时满足时才加入。控制面不接触密钥；零知识 E2E 内容拒绝进入本期。
6. 每条 join、submit、resume、presence、确认、冲突和撤权消息显式携带完整四段内容身份。`sourceEditorId`、session generation 和 connection ID 只作局部生命周期字段，不能替代内容身份。
7. 后端采用 Nest 原生 `@Module`、`@Injectable`、Guard、Pipe、Interceptor、custom metadata 和 DI provider；声明式装配是能力发现的唯一事实源，业务顺序仍由显式 use case/service 处理。

## 3. 数据所有权与链路

### 3.1 正文操作链路

```text
Protyle semantic event
  -> React collaboration store creates operation envelope
  -> dedicated WSS ingress parses protocol contract
  -> session capability and four-part identity are checked at ingress
  -> CollaborationCoordinator orders/idempotently accepts the envelope
  -> KernelCollaborationPort submits semantic operation to Go Kernel
  -> Go semantic reducer + content transaction produce canonical result
  -> Kernel history append + current content update
  -> coordinator broadcasts canonical operation to authorized sessions
  -> Protyle host applies the semantic result
```

React 不复制正文；Zustand 只保存会话状态、未确认操作引用、冲突摘要和 presence。Protyle/Kernel 仍是正文消费点。迟到结果必须带原始四段身份和 session generation，不能回填当前页面的身份。

### 3.2 权限和撤权链路

```text
L2 DocumentAccessPolicy / AccessChanged
  -> CollaborationAccessGuard / SpaceConnectionRegistry
  -> session capability invalidation
  -> dedicated WSS close + no further broadcast
```

`DocumentAccessPolicy` 是 ACL 唯一 owner。Guard 只在真实协作入口调用它；协调器消费已授权 capability 和撤权代次，不重新解释用户组、角色或 deny/allow。撤权事件先以后端提交事实为准，再关闭受影响连接。

### 3.3 加密内容链路

```text
authorized encrypted-library session
  -> existing content-key admission in Kernel boundary
  -> Gateway passes opaque service-authenticated operation bytes
  -> Kernel decrypts/applies semantic operation in content session
  -> canonical history/result returns without key or plaintext logging
```

密钥不进入 Nest service、Prisma、审计、日志或浏览器持久化。若 Kernel 无法在授权内容会话内取得密钥，Ingress 以 `encrypted-collaboration-unavailable` 结束，不走明文、快照或备用通道。

## 4. 模块边界与文件归属

| 顺序 | 模块 owner | 生产范围 | 上下游合同 |
| --- | --- | --- | --- |
| 1 | L3.1 contracts | `enterprise/packages/contracts/src/realtime-collaboration.ts`、聚合导出、OpenAPI/协议 schema | 四段身份、操作、确认、冲突、受限加密错误码冻结后交给所有模块 |
| 2 | Go Kernel production bridge | `kernel/collab/**`、Kernel history/transaction bridge、Go 集成夹具 | 消费已解析语义操作，返回 canonical result；不读 HTTP/DOM，不建第二正文模型 |
| 3 | Nest collaboration control | `enterprise/apps/api/src/collaboration/**`、模块 metadata、Guard、coordinator、WSS handler | 认证、ACL、开关和审计；调用 Kernel port，不复制 reducer |
| 4 | Collaboration transport | `enterprise/apps/api/src/kernel/**` 中专用协作 upgrade/connection owner；不改 Protyle push route | 专用 WSS 双向操作；浏览器数据帧只能进入协作合同 |
| 5 | React/Protyle host | `enterprise/apps/web/src/**` 协作 session store、Host action、shadcn variants | Zustand 只保存会话/冲突/presence；正文仍经 Protyle semantic action |
| 6 | Integration owner | 根命令、永久 contract/integration/browser/E2E、文档收口 | 全部模块完成后统一接入唯一 aggregate，不抢写共享合同 |

共享文件（contracts 聚合器、Prisma schema、CoreModule、WSS upgrade 主入口、根 `package.json` 和唯一 runner）由 integration owner 统一修改。当前无子代理并行；共享合同和单副本生命周期有强顺序依赖，按上表串行集成。

## 5. 单副本运行模型

- API 进程内 `CollaborationConnectionRegistry` 按四段身份和 session generation 管理 pending/active/closed 连接；不保存正文快照。
- 每个文档最多一个活动 coordinator；同一 API 副本内的并发提交通过显式 use case 顺序化，Kernel 仍是最终提交 owner。
- presence 仅内存 TTL；API 重启、ACL 撤权、加密密钥失效或功能关闭都会清除会话并要求重新加入。
- operation log 不写 PostgreSQL；Go Kernel 的 append-only history 是恢复事实。API 只保留必要的会话、审计、开关和指标元数据。
- 首期不引入 Redis、NATS、BullMQ 或其他跨副本消息总线；未来多副本必须另立 ADR，不在本期留下兼容分支。

## 6. 声明式 Nest 装配

### 6.1 声明目标

- `CollaborationModule` 通过 `@Module` 声明 provider、Guard、Interceptor、Gateway adapter 和 Kernel port。
- `@CollaborationOperationHandler(kind)` 只声明操作类型、版本和能力要求；DiscoveryService 在模块启动时发现 provider。
- handler 的优先级、冲突和作用域由公开 metadata 合同决定；同一 `kind + version` 多重 provider 使应用启动失败，不使用中央 `switch` 或文件名约定。
- provider 默认 singleton；连接状态、presence 和文档 coordinator 由显式 session registry 管理，不通过隐式全局变量保存。

### 6.2 声明不承担的职责

装饰器和 metadata 只负责装配、能力描述和观测标签；因果排序、事务、冲突处理、加密 admission 和错误转换仍由 `CollaborationCoordinator`、`KernelCollaborationPort` 和 `CollaborationAccessGuard` 等显式 service/use case 承担。不得把网络 I/O、fallback 或复杂控制流塞进装饰器工厂。

### 6.3 启动和失败语义

- 缺少任何必要 operation handler、同义版本冲突或 Kernel port 未绑定时，应用启动失败，不以空 registry 或备用 handler 继续服务。
- `DiscoveryService` 只发现 `CollaborationModule` scope 内 provider；不扫描源码、不维护第二份名称列表。
- 运行时未知操作、过期版本或能力不足返回稳定拒绝码，保留完整原始异常 stack 和 request/session 关联。

## 7. 协议与状态合同

所有消息复用 L3.0 合同并增加生产字段 `sessionGeneration` 和 `featureMode`，不增加同义身份字段：

```ts
type CollaborationFeatureMode = "standard" | "restricted-encrypted";

interface ProductionOperationEnvelope {
  identity: DocumentIdentity;
  operationId: string;
  clientId: string;
  clientSequence: number;
  causalContext: VersionVector;
  sessionGeneration: number;
  operation: CollaborationOperation;
}
```

`featureMode` 属于 join/ready 会话合同；操作 envelope 只带由会话绑定的 identity 和 generation，不把同一模式再次塞入每条操作。所有拒绝结果带 `identity + operationId + code`，不返回整篇正文。

## 8. 数据库与审计边界

- Prisma/PostgreSQL 只新增必要的协作开关、会话元数据和审计投影；不新增正文、CRDT 文档快照或 operation payload 表。
- ACL、开关和撤权使用既有事务/`AccessChanged` 合同；连接关闭发生在提交事实之后。
- 审计记录 `operationId`、四段身份摘要、actor/session、结果、冲突码和耗时；不记录正文、密钥、完整 payload 或 token。
- 需要跨重启恢复的 canonical operation 与历史由 Go Kernel 维护；控制面审计只证明动作发生，不成为内容回放事实。

## 9. 关键函数中文备注要求

实现必须为以下跨边界函数添加必要中文备注，说明前置条件、后置状态和副作用：

- 协作 WSS upgrade/activate：说明 Origin、Session、ACL 重验和 pending 到 active 的原子边界。
- `submitOperation`/coordinator dispatch：说明四段身份、generation、幂等顺序和不得发送正文快照的约束。
- Kernel bridge/apply：说明 reducer 是唯一正文状态转换、history append 与当前内容更新的提交关系。
- ACL revoke/connection close：说明提交后撤权、关闭旧连接和禁止迟到广播的顺序。
- encrypted admission：说明密钥只在 Kernel 内容会话内可用，失败只能返回受限错误。
- resume/replay：说明只读取 canonical history 缺口，不回填页面身份或切换备用路径。

## 10. 诊断与异常可观测性

使用既有 logger/trace 机制保留稳定标签：`collaboration.join`、`collaboration.operation`、`collaboration.resume`、`collaboration.conflict`、`collaboration.revocation`、`collaboration.lifecycle`。每个 catch、Promise rejection、WS error 和 Kernel bridge 转换记录原始 `name/message/stack`、requestId/sessionGeneration、operationId 和四段身份摘要；敏感信息只做字段级脱敏，不用 message 覆盖 stack。

热路径只记录采样后的耗时、结果和计数，不记录正文或密钥；达到容量/频率上限时记录限流原因和当前计数，便于区分协议拒绝、Kernel 不可用和 ACL 撤权。

## 11. 设计模式和简化比较

- **Command**：语义操作可记录、重放和生成 inverse；避免整篇正文快照。
- **State**：只用于连接和会话生命周期；避免布尔字段组合出 revoked/reconnecting。
- **Observer**：仅用于 canonical broadcast 和 ephemeral presence；订阅者不拥有正文。
- **Dependency Injection + Plugin metadata**：声明 operation handler 和 Kernel port，新增操作不改中央 switch。
- **Bridge**：仅保留真实跨语言内容协议转换的 Kernel bridge；它不是同形 DTO 透传。

更简单的“HTTP transaction 后广播”无法表达并发文本/结构语义；“浏览器 CRDT + PostgreSQL 快照双写”会产生第二正文事实源；“多副本 + Redis/NATS”超出已批准首期部署并增加恢复状态。因此单副本、Kernel canonical history 和专用 WSS 是当前最短数据路径。

## 12. 测试矩阵与唯一门禁

L3.1 是一个集中验证大阶段。全部生产代码、旧路径清理和 code-review/test-governance 复评完成后，已一次性执行 `cd enterprise && pnpm verify:l3-production`；结果见 [L3.1 验证报告](../verification/l3.1-realtime-collaboration.md)。

| 稳定合同 | 最低层级 | 真实/模拟边界 | 主要故障 |
| --- | --- | --- | --- |
| 四段身份、操作、冲突和加密拒绝码 | contract | Zod/TypeScript 标准 runner | 字段缺失、同义字段或错误码漂移 |
| Go reducer/bridge、history append/replay、inverse | Go unit/integration | 真实 Kernel 内容模型和隔离内容目录 | 块/AV/历史语义损坏 |
| WSS join/submit/resume、ACL 撤权和单副本重启 | API integration | 真实 Nest/WS、固定 PostgreSQL、受控 Kernel | 迟到写入、旧 grant 推送、假恢复 |
| encrypted admission、日志脱敏和功能开关 | security/integration | 真实授权链，密钥使用由 Kernel 边界提供 | 密钥泄露或明文降级 |
| React Protyle host、Zustand 会话、冲突和只读状态 | component/browser integration | 真实组件；外部 Kernel 可明确替换 | 页面身份错绑、状态覆盖、移动端溢出 |
| 单副本生命周期和性能上限 | integration/performance | 固定客户端数、操作大小和消息速率 | 无限排队、资源泄露 |
| 生产路径隔离、无旧 WS 写入、无 PostgreSQL 正文表 | static/AST/config | 结构化依赖和配置审计 | 原型/生产边界泄漏 |

统一 runner 必须支持独立 case、标准 hook/cleanup、按名称过滤和结构化报告；不得新增逐文件子进程、顶层 assert、源码字符串行为证明或完整内部 mock 链。

## 13. 完成定义

- L3.1 PRD、架构、ADR、实施计划状态一致并指向同一首期边界。
- 生产 React/Protyle、Nest WSS、Go Kernel bridge/history、ACL 撤权、受限加密、单副本重启和灰度开关全部闭合。
- 没有第二正文事实源、整文档覆盖、LWW、锁、隐式身份推断、明文降级、跨副本兼容分支或旧 Protyle WS 写入。
- `pnpm verify:l3-production` 集中通过，固定测试数据库按约定保留，文档和目标状态已更新为 `verified`；功能开关仍默认关闭，必须经过灰度门禁后才允许开放。
