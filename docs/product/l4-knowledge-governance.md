## 1. 目标与边界

### 1.1 本期包含

- 文档生命周期：`draft`、`in-review`、`approved`、`published`、`archived`、`rejected`。
- 可配置审批流：指定审核人或用户组，支持提交、通过、退回和重新提交；每次决策可追溯。
- 页面验证：文档负责人或审核人设置验证周期，页面显示 `verified`、`needs-review` 或 `expired` 状态。
- 文档模板：管理员维护带初始结构和治理默认值的模板，创建文档时生成一次性初始内容；模板不是第二正文事实源。
- 密级分类：组织或空间定义分类级别，文档只能继承或提升级别，不能绕过上游策略降低级别。
- 生命周期归档：按空间策略识别长期未更新内容，归档前通知负责人；归档内容可在授权范围内恢复。
- 保留策略与法律保留：管理员配置保留期限；法律保留冻结归档清理，策略变更和解除均进入审计。
- 导出与分享水印：导出 PDF、只读分享和下载结果按当前用户、组织、空间、文档和时间生成可追溯水印；水印结果不写回正文。
- 治理仪表盘：展示待审批、即将过期、已过期、待归档、法律保留和策略执行失败的治理队列。

### 1.2 已确认的企业能力范围

以下能力均纳入 L4，按依赖分为四条产品线，不再重复讨论是否建设：

| 产品线 | 能力 | 用户结果 |
| --- | --- | --- |
| 身份与机器访问 | MFA、SAML、SCIM、API Key | 员工可用企业身份登录，离职/调岗能及时收回，自动化客户端可被单独审计 |
| 知识治理 | 页面验证、审批流、模板、密级、归档、保留、法律保留、水印 | 知识有负责人、状态、期限和可追溯外发记录 |
| 内容范围与嵌入 | 个人空间、跨空间搜索、PDF 企业导出、Draw.io、Excalidraw | 用户能在授权范围内管理个人草稿、统一发现内容并保留图表编辑能力 |
| 智能辅助 | AI Chat | 用户可在授权范围内提问并得到可追溯引用；无权限内容不可被回答或侧信道泄露 |

个人空间、跨空间搜索和 AI Chat 仍必须继承组织/空间/文档四段身份及 ACL，不创建绕过权限的全局内容池。

### 1.3 本期不包含

- 多副本 Gateway、跨区域实时协作、调度和灾备；这些属于 L5。
- 移动端生产体验、离线无限编辑和零知识端到端协作；它们需另立方案。
- AI 对话、权限感知问答、摘要和知识关联；这些属于 L6。
- 计费、商业订阅、云托管 entitlement 和 Docmost 专属授权系统。
- 全量 DLP、OCR、外部 SIEM 编排和法律法规自动判定；本期只提供可配置治理事实和稳定审计事件。
- 直接修改正文内容模型、把治理状态写入 `.sy` 正文或在 PostgreSQL 保存正文快照。

## 2. 对标输入与能力差距

### 2.1 同类调研报告

| 案例 | 交互方式 | 优点 | 局限 | 奇点借鉴 |
| --- | --- | --- | --- | --- |
| Docmost Enterprise | 空间内权限、企业身份、页面验证/模板等以管理入口集中配置 | 管理员路径短，空间边界清楚 | 企业能力按版本分层，跨空间治理需要额外约束 | 复用空间作为主要治理边界，新增组织级策略和审计 |
| Confluence | 页面状态、审批、模板和空间权限组合，治理信息靠页面侧栏呈现 | 复杂组织可逐步启用，状态可见性成熟 | 配置项多，权限继承容易让用户困惑 | 用少量明确状态和服务端解释减少隐式继承 |
| Outline | 集合/文档权限、公开分享和全文搜索 | 阅读体验简洁，分享链路清晰 | 企业身份和合规治理需要外部系统补足 | 保持思源风格的轻量阅读面，企业能力集中在管理面板 |
| GitLab Wiki / Handbook | Markdown 文档、审阅和代码式变更记录 | 变更可追溯、审阅责任明确 | 非技术用户的状态和审批反馈较弱 | 将版本/审批/验证记录转成非技术用户可读的时间线 |

调研结论：L4 采用“空间内执行、组织级策略、文档侧状态、统一审计”的组合；复杂身份协议只影响登录和成员生命周期，不直接改文档权限；AI 和跨空间搜索必须在同一授权查询边界内执行。

本节只记录可复核的官方仓库输入和本仓实现事实，不把 Docmost EE 私有实现推断成公开合同。

| 官方 Docmost 输入 | 奇点当前状态 | L4 处置 |
| --- | --- | --- |
| 实时协作、空间、权限、组、评论、历史、搜索、附件、嵌入和图表 | L1-L3 已实现或由 Kernel 提供，L3.1 已验证 | 保持现有边界，不在 L4 重做 |
| EE `page-verification` | 当前没有独立的文档验证周期和治理队列 | 纳入 L4 |
| EE `template` | Kernel 有内容模板语义，但没有企业模板目录、管理员发布和治理默认值 | 纳入 L4 |
| EE `mfa`、`scim`、`api-key` | 当前 OIDC/本地会话、组织成员和服务认证存在，但没有对应完整产品合同 | L4 只记录差距，另立身份安全计划 |
| EE `personal-space` | 当前模型以组织/空间为主，没有个人空间产品边界 | 后续产品评审，不混入 L4 治理 |
| EE `ai-chat` | 当前没有 AI 知识库产品合同 | 归入 L6 |
| EE `pdf-export` | Kernel 已有导出能力，但治理水印和导出审计尚未闭合 | L4 增加治理水印与审计，不重写导出内核 |
| EE `saml` / `scim` / `api-key` | 当前有 OIDC、本地会话和服务认证，但缺少企业身份生命周期产品合同 | L4 增加身份安全产品合同，协议实现另立架构边界 |
| EE `personal-space` / 跨空间发现 | 当前模型以组织/空间为主，没有个人空间和授权聚合查询 | L4 增加个人空间和授权范围内跨空间搜索 |
| EE `drawio` / `excalidraw` / `ai-chat` | 当前没有统一的嵌入和授权问答产品合同 | L4 增加外部内容嵌入和引用式问答边界 |

官方输入来源：Docmost [README Features](https://github.com/docmost/docmost#features) 和 EE 目录中的 `apps/client/src/ee` 模块清单。奇点实现事实以 L1-L3 验证报告为准。

## 3. 角色与用户故事

### 3.1 角色

- **作者**：创建文档、提交审批、响应退回意见并处理验证提醒。
- **审核人**：在授权范围内通过或退回文档，并留下结构化决策记录。
- **空间管理员**：配置空间模板、密级、审批流、验证周期、保留策略和归档策略。
- **合规管理员**：设置组织级治理规则、法律保留和水印策略，查询治理审计。
- **阅读者**：只能看到当前授权范围内的生命周期状态、验证状态和水印后的导出结果。
- **运维人员**：处理策略执行失败、Worker 停止和容量告警，不绕过业务状态直接改数据库。

### 3.2 用户故事

- 作为作者，我可以把文档提交给指定审核人，知道当前处于待审、通过还是退回状态。
- 作为审核人，我可以查看文档当前版本和治理摘要，批准或退回，并让决定进入审计。
- 作为空间管理员，我可以用模板创建新文档，并让模板携带默认密级、负责人和验证周期。
- 作为合规管理员，我可以设置保留期限和法律保留，确保到期内容不会被自动清理。
- 作为阅读者，我可以看到文档是否已验证或过期；导出的文件带有当前访问者可追溯水印。
- 作为运维人员，我可以看到归档、验证和保留任务的失败原因，并重试同一任务而不重复改变状态。

### 3.3 企业能力用户故事

- 作为员工，我可以用企业 SAML 登录并完成 MFA；管理员禁用账号后，既有会话和 API Key 在约定时间内失效。
- 作为身份管理员，我可以通过 SCIM 创建、更新、停用成员和用户组，看到同步结果；同步不会越权修改文档 ACL。
- 作为自动化客户端负责人，我可以创建带用途、过期时间和最小权限的 API Key，并只看到一次完整密钥。
- 作为用户，我可以在个人空间保存草稿，并在已授权的多个空间中搜索结果；无权内容不出现在数量、摘要、片段或 AI 引用中。
- 作为用户，我可以编辑 Draw.io/Excalidraw 内容并在文档中看到稳定预览；嵌入失效时文档正文仍可读且能明确恢复。
- 作为用户，我可以向 AI Chat 提问并打开每条引用对应的文档位置；权限撤回后，新的回答不再使用该内容。

## 4. 交互与状态

### 4.1 文档治理面板

文档面板显示当前生命周期、审批状态、验证状态、密级、负责人、下次验证时间和保留状态。状态由服务端治理事实提供，前端不能通过本地状态解除审批、密级或保留限制。

### 4.2 审批流

1. 作者选择当前版本并提交审批；提交后文档进入 `in-review`，正文仍由 Kernel 管理。
2. 服务端确认作者具有编辑能力、当前版本未变化、审批策略可用后创建待审事项。
3. 审核人只能对当前版本做一次明确决策；通过进入 `approved` 或 `published`，退回进入 `rejected` 并要求重新提交。
4. 文档内容发生新变更后，已通过的审批不自动覆盖新版本；新版本需要按策略重新提交。

### 4.3 验证与归档

- 验证周期到达前提醒负责人；到期进入 `needs-review`，超过宽限期进入 `expired`。
- 归档候选必须满足空间策略、更新时间和无活动审批条件；系统先通知负责人，再执行归档。
- 法律保留存在时，归档内容不得清理；解除法律保留必须留下审计记录并重新计算保留日期。
- 恢复归档文档不自动变为已发布或已验证，必须重新遵循当前治理策略。

### 4.4 水印和导出

- 水印应用于导出/分享响应边界，包含组织、当前用户、文档标识摘要、访问时间和策略声明。
- 水印不得写入 `.sy` 正文、Kernel 历史、PostgreSQL 正文表或浏览器持久化存储。
- 无法取得授权身份或水印策略时，导出明确失败，不返回无水印备用结果。

### 4.5 身份安全

- SAML 登录失败、MFA 失败、SCIM 同步失败和 API Key 过期都显示稳定的用户结果与关联编号，不暴露协议响应正文或密钥。
- SCIM 的成员/组变更进入组织审计；停用成员会触发会话、协作连接和机器凭据的生命周期收敛。
- API Key 只用于机器身份，界面不允许把它当作用户登录凭据；密钥明文只在创建成功时显示一次。

### 4.6 个人空间、跨空间搜索与嵌入

- 个人空间默认仅本人可见；组织管理员的可见性、接管和保留动作必须有明确策略与审计。
- 跨空间搜索先选择组织和允许的空间集合，再返回文档、摘要和高亮；换空间、撤权或迟到响应不得覆盖当前结果。
- Draw.io/Excalidraw 以文档嵌入对象呈现，编辑器故障只影响嵌入状态，不阻断正文读取；导出和审计明确记录嵌入版本。

### 4.7 AI Chat

- 每次问题、回答和引用都绑定当前组织、空间、笔记本和文档授权上下文；回答只使用当时可读内容。
- 权限变更、文档归档或法律保留变化后，新的回答必须重新检查授权；历史回答保留引用状态，不伪造为当前可读。
- AI 服务不可用、引用验证失败或内容密级不允许时，显示可重试的明确结果，不返回无引用的“可能答案”。

## 5. 验收标准

| 编号 | 验收标准 | 失败风险 | 最低充分证据 |
| --- | --- | --- | --- |
| L4-LIFE-01 | 文档生命周期只能按定义状态转移，非法顺序明确拒绝 | 审批绕过、状态回退 | API integration + state transition |
| L4-APP-01 | 作者提交当前版本后产生唯一待审事项，重复提交幂等 | 重复审批、版本错配 | HTTP/DB integration |
| L4-APP-02 | 审核人通过或退回后，结果、意见和当前版本可追溯 | 决策丢失 | API + audit integration |
| L4-APP-03 | 新正文版本不会继承旧版本的批准结果 | 旧批准覆盖新内容 | Kernel/version integration |
| L4-VER-01 | 验证提醒、到期和过期状态按策略产生且可恢复 | 知识过期无人知晓 | Worker + API integration |
| L4-TPL-01 | 只有授权管理员能发布模板；创建文档只生成一次初始内容 | 模板越权、正文双写 | API + browser integration |
| L4-CLS-01 | 文档只能继承或提升密级，降低密级被拒绝 | 敏感内容降级泄露 | policy contract + integration |
| L4-RET-01 | 保留策略到期前不清理内容，法律保留覆盖自动清理 | 合规数据误删 | Worker + DB integration |
| L4-RET-02 | 归档/恢复/清理任务幂等，失败保留完整堆栈和可重试状态 | 重复清理、孤儿任务 | lifecycle integration |
| L4-WM-01 | PDF/分享/下载结果按当前授权身份生成水印 | 内容外发不可追踪 | real HTTP/export integration |
| L4-WM-02 | 水印不进入正文、历史、控制面或浏览器持久化 | 第二事实源、敏感泄露 | storage/logger audit |
| L4-AUD-01 | 治理决策、策略变化和任务结果进入现有审计链 | 无法追责 | audit integration |
| L4-ACL-01 | 治理动作复用现有 ACL owner，viewer 不能审批/改策略 | 客户端自证权限 | real HTTP/DB integration |
| L4-OBS-01 | 治理任务和 API 异常包含原始 `name/message/stack` 与关联 ID | 无法排障 | logger integration |
| L4-ROLL-01 | 关闭治理开关后新治理动作被拒绝，在途任务收敛且不伪造成功 | 灰度回滚失控 | lifecycle integration |
| L4-ID-01 | SAML + MFA 登录成功/失败状态稳定，禁用成员后会话按策略失效 | 账号绕过、会话残留 | identity integration + browser |
| L4-ID-02 | SCIM 用户和组同步可重试、幂等，不能直接改变文档 ACL | 外部权限误写内容权限 | SCIM contract + integration |
| L4-ID-03 | API Key 生命周期按用途、过期和最小权限校验，完整密钥只显示一次且不进日志；验证服务可供机器入口接入 | 机器凭据泄露 | API + logger audit |
| L4-SPACE-01 | 个人空间默认隔离，管理员策略动作可追溯 | 私人草稿泄露或无法接管 | access + audit integration |
| L4-SEARCH-01 | 跨空间搜索只返回当前授权结果，迟到响应不能覆盖当前空间 | 串库、侧信道泄露 | API + browser + race test |
| L4-EMBED-01 | Draw.io/Excalidraw 嵌入可编辑、可预览，失效不阻断正文 | 外部嵌入破坏文档可读性 | browser + export integration |
| L4-AI-01 | AI Chat 回答带可验证引用，授权变化后不使用失权内容 | 越权回答、无法追溯 | policy + retrieval integration |
| L4-AI-02 | AI 不可用或引用验证失败时稳定拒绝，不返回无依据答案 | 误导性内容 | API + browser |

## 6. 约束

- Go Kernel 继续拥有正文、块、AV、历史和导出内容事实；Nest/PostgreSQL 只拥有治理元数据、策略、任务和审计。
- 每条治理请求、事件和任务显式携带 `organizationId + spaceId + notebookId + documentId`；不从路径、DOM、首个响应或模板名称推断身份。
- ACL 仍由 `DocumentAccessPolicy` 唯一负责；治理模块消费 capability，不复制角色算法。
- 新增 API 使用 Nest 声明式模块、Guard、Pipe、Interceptor、custom metadata 和 DI provider；生命周期状态转移由显式 service/use case 拥有。
- 所有用户可见文案走现有 i18n；页面使用现有 React/shadcn/Tailwind 设计系统，不创建局部视觉事实源。
- 功能开关默认关闭；L4 先在非生产和单组织灰度，不能改变 L3 单副本合同。
- SAML、SCIM、MFA、API Key 的密钥、断言、token 和 AI 凭据不进入正文、普通审计正文或浏览器持久化；日志只保留脱敏关联信息。
- 跨空间搜索和 AI retrieval 必须以服务端授权过滤为首个事实边界；前端过滤不能作为安全控制。
- 外部 Draw.io/Excalidraw 内容按不可信输入处理，加载失败、回源失败和版本不匹配都必须可恢复。

## 7. 阶段门禁

L4 只有在本 PRD、架构方案、ADR 和实施计划完成评审后进入 implementation。全部生产代码、迁移、调用方、永久测试、旧路径清理和文档完成后，统一进行 code-review + test-governance，再执行唯一 `pnpm verify:l4-governance` aggregate。
