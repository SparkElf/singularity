# 奇点 L3.0 实时协作语义原型产品需求

> L2 已完成异步协作。L3.0 只验证实时协作是否能保持思源块语义和历史语义，不把未经验证的 CRDT、锁或最后写入获胜方案接入生产编辑器。

## 1. 目标与阶段边界

L3.0 的目标是建立一个可重复运行的独立语义原型：两个或更多受控客户端同时编辑同一文档时，合并结果保持块身份、引用关系、嵌入关系、属性视图数据、历史和撤销语义。原型必须能证明失败时停止在原型边界，不会悄悄退化为整篇文档覆盖或最后写入获胜。

L3.0 是产品化前的技术门禁，产物不是企业生产功能。只有所有 L3.0 验收项通过，才可进入 L3.1 生产候选设计；L3.1 仍需独立评审，不由 L3.0 自动授权。

### 1.1 本阶段包含

- 两个或多个受控客户端同时打开同一文档，并观察远端变更和自己的确认状态。
- 普通段落、标题、列表/容器块的并发文本与结构操作。
- 块引用和嵌入块的目标身份保持；引用目标被移动、删除或重命名时不按文本近似迁移。
- 属性视图中的单元格并发编辑，以及属性视图结构未被破坏时的可见结果。
- 以操作因果顺序形成可追踪历史；本地撤销只撤销当前客户端自己的已提交操作，不撤销其他客户端后续操作。
- 临时在线状态与光标/选区投影；它们不是正文事实源，不进入内容历史。
- 短暂断线后的会话重新建立、未确认操作的明确结果和重复操作不产生二次内容变更。
- 每条原型消息显式携带 `organizationId + spaceId + notebookId + documentId`；测试夹具也不能省略任一身份。
- 复用 L2 文档 ACL 的授权事实；权限撤销后，原型会话停止接受新操作并清除远端推送。

### 1.2 本阶段不包含

- 生产 Protyle 接入、生产 Gateway 路由、生产数据库迁移或默认启用实时协作。
- 把正文复制到 PostgreSQL、把 CRDT 文档作为第二内容事实源，或在 Kernel 外保存可独立恢复的正文快照。
- 锁住整篇文档、最后写入获胜、整篇文档覆盖、客户端互相发送未经服务端确认的正文快照。
- 邮件/短信等外部通知；L2 站内通知继续使用既有异步收件箱。
- 跨文档原子编辑、附件二进制并发、插件私有状态同步、复杂属性视图结构迁移、移动端生产体验和离线无限编辑。
- 未经独立安全评审的端到端加密协作协议；加密内容只验证显式内容身份和锁态拒绝合同。

## 2. 本地先例与决策依据

本阶段不进行非必要网页搜索。仓库已有事实足以定义产品门禁：

| 本地来源 | 已确认事实 | L3.0 借鉴 | 不直接照搬 |
| --- | --- | --- | --- |
| [ADR-006](../adr/0006-realtime-collaboration-gate.md) | WebSocket 广播不等于 CRDT；块 ID、引用、属性视图、历史和撤销必须先无损验证 | 把语义原型作为生产前置门禁 | 不把现有推送通道包装成实时协作 |
| [Protyle 浏览器运行时方案](../architecture/protyle-browser-host.md) | Protyle 拥有编辑器 DOM、块事务、快捷键和内容插件；Host 只消费类型化事件 | 原型必须以语义操作和公共身份合同测试，不扫描 DOM 推断事实 | 不在 React store 复制编辑器正文 |
| [L2 异步协作 ADR](../adr/0029-l2-async-collaboration-boundary.md) | Go Kernel 拥有正文与历史，控制面拥有 ACL、评论、通知；内容身份必须显式传递 | 复用四段身份和服务端 ACL 结果 | 不把 L3 临时状态写进 L2 评论/通知表 |
| 当前 Go Kernel HTTP/WS 合同 | 正式写入走 HTTP，WebSocket 只接收推送 | 原型必须独立验证双向操作协议和拒绝语义 | 不修改现有生产 WS 为“准 CRDT” |

Yjs、Automerge 或其他 CRDT 引擎只能作为架构阶段的候选实现。产品验收只约束用户可观察的合并、身份、历史和撤销结果，不提前把某个库名称写成产品合同。

## 3. 角色与用户故事

### 3.1 角色

- **编辑者**：拥有 L2 `editor` 能力，可在原型中提交正文操作并看到其他编辑者的已确认变更。
- **观察者**：拥有 L2 `viewer` 或 `commenter` 能力，只接收允许展示的远端内容和临时状态，不产生正文操作。
- **空间管理员**：可撤销文档访问；撤销后所有原型客户端都必须停止接受该文档的新操作。
- **原型维护者**：运行固定夹具和故障注入，确认语义门禁通过或明确停止产品化。

### 3.2 用户故事

- 作为两名编辑者，我可以同时改同一段文字，双方最终看到相同且不丢失的结果。
- 作为编辑者，我可以在文档结构变化时继续使用真实块引用和嵌入，引用不会因为标题或 DOM 位置变化而指向错误目标。
- 作为编辑者，我可以修改属性视图单元格，另一个客户端的并发单元格修改不会把属性视图写成不可解析数据。
- 作为编辑者，我撤销自己的操作时，不会撤销其他人的操作；刷新后历史仍能解释当前结果。
- 作为观察者，我可以看到允许的远端结果，但不能通过浏览器状态或伪造消息提交正文操作。
- 作为空间管理员，我撤销权限后，旧客户端不能继续写入或接收该文档推送，恢复权限后也必须重新授权。
- 作为原型维护者，我可以在断线、重复消息、乱序消息和迟到消息下得到明确的拒绝或收敛结果，而不是静默降级。

## 4. 交互与状态定义

### 4.1 加入文档协作

1. 客户端从当前 L2 空间 Session 得到四段内容身份和最新 ACL 能力。
2. 服务端确认 `editor`/`viewer` 能力和当前文档协作版本后建立原型会话。
3. 客户端先展示本次会话的 canonical 文档状态，再接收带因果信息的远端操作；不能从第一个远端响应推断笔记本或文档身份。
4. 会话状态只能是 `connecting`、`ready`、`reconnecting`、`revoked`、`closed`；状态转换必须可观察，不能用普通网络失败伪装成撤权。

### 4.2 并发编辑

- 每次正文变更都以最小语义操作提交，操作带客户端来源、单调序号、因果上下文和四段内容身份。
- 服务端确认后向相关客户端广播 canonical 操作或等价增量；客户端不广播未经确认的最终正文快照。
- 同一逻辑结果在不同消息到达顺序下必须收敛；无法收敛时显示明确原型失败并停止产品化，不使用静默 LWW。
- 块 ID、引用目标 ID、嵌入目标 ID 和属性视图/单元格 ID 是稳定语义字段；不得按文本、路径、数组位置或首项推断。

### 4.3 历史与撤销

- 历史记录的是已确认的语义操作批次及其因果关系，不记录光标、选区或 presence。
- “撤销”默认表示当前客户端撤销自己最近的可撤销操作；如果该操作已被其他客户端依赖，系统仍必须保持结构合法并明确显示结果。
- 旧历史不可变；重放同一操作不会产生第二次内容变化，刷新/重新加入后结果与实时会话一致。

### 4.4 断线、撤权与恢复

- 断线进入 `reconnecting`，期间禁止提交未经确认的正文操作；重新授权成功后按服务端版本重新同步。
- 重复、乱序或过期操作必须由唯一协议边界拒绝或幂等吸收，并留下完整异常堆栈和会话关联信息。
- ACL 撤销进入 `revoked`，立即停止新写入和远端推送，清除本地实时会话；不自动切换到其他空间、文档或备用通道。

## 5. 验收标准

每条标准均以用户结果或跨层稳定合同判定；源码字符串、私有方法、固定截图或内部 mock 不能单独作为通过证据。

| 编号 | 验收标准 | 主要风险 | 最低充分证据 |
| --- | --- | --- | --- |
| L3-SEM-01 | 两个编辑客户端对同一普通块进行并发插入、删除和替换后，双方最终得到相同结果且无已确认操作静默丢失 | LWW 或覆盖写 | 语义原型 integration + browser prototype |
| L3-SEM-02 | 普通块、标题、列表/容器块的父子关系和稳定 block ID 在并发结构操作后仍可解析 | 树结构损坏 | Kernel/语义模型 integration |
| L3-SEM-03 | 块引用和嵌入始终保留真实目标 notebook/document/block 身份；目标变化只产生明确失效结果，不自动近似迁移 | 跨库引用或错误重锚 | contract + integration |
| L3-SEM-04 | 属性视图单元格并发编辑后，视图结构、行/列 ID 和单元格值仍可解析；不产生第二套 AV 事实源 | AV 数据损坏 | Go Kernel/AV integration + prototype |
| L3-HIS-01 | 已确认操作形成可重放历史；重新加入或刷新后的文档结果与实时会话一致，旧历史不可变 | 历史覆盖或重放漂移 | operation-log integration |
| L3-HIS-02 | 客户端撤销只影响自己的可撤销操作，不撤销其他客户端后续操作；撤销失败有明确结果且不静默成功 | 误撤销他人内容 | multi-client integration |
| L3-ID-01 | 每条操作、确认、订阅和恢复消息都显式携带 organizationId、spaceId、notebookId、documentId；任何缺失身份都在协议入口失败 | 跨空间/跨库污染 | contract + protocol integration |
| L3-ID-02 | 迟到、乱序和重复消息不能覆盖另一文档/空间代次；被拒绝消息不会改变正文 | 竞态覆盖 | deterministic protocol test |
| L3-ACL-01 | viewer 不能产生正文操作；ACL 撤销后旧会话不能继续写入或接收该文档推送 | 客户端自证权限 | HTTP/WS integration |
| L3-PRES-01 | presence、光标和选区只作为临时投影，不进入正文历史或恢复结果；客户端销毁后不残留 | 状态污染正文 | browser/component prototype |
| L3-REC-01 | 断线重连不会重复提交已确认操作；恢复失败进入明确 closed/reconnecting 结果，不切换备用地址或伪造成功 | 重复写和假恢复 | fault-injected integration |
| L3-GATE-01 | 固定语义夹具和故障矩阵可由标准 runner 重复执行；任一关键语义失败都阻止 L3.1 生产候选 | 只验证 happy path | aggregate prototype runner |

## 6. 约束

- **事实源**：Go Kernel 继续拥有正文、块和历史；L3.0 原型使用隔离测试内容，不向生产 PostgreSQL 写正文。
- **身份**：所有消息显式携带四段内容身份；`sourceEditorId` 只标识内存编辑器实例，不替代文档身份。
- **权限**：服务端/协议入口是 ACL owner；客户端 UI 的只读状态不能作为授权依据。
- **一致性**：禁止整文档快照覆盖、锁、LWW、静默丢弃或隐式 fallback；失败必须可观察并停止阶段推进。
- **性能**：原型至少记录操作确认延迟、合并耗时、重连恢复耗时和消息大小；不以无限队列或无限历史掩盖性能问题。
- **安全**：操作和 presence 不记录正文之外的敏感凭据；日志保留原始异常 `name/message/stack`，避免正文、令牌和完整 payload。
- **兼容**：沿用 React 19、Vite 8、Tailwind 4、shadcn、NestJS、Prisma 和 Go Kernel；不在 L3.0 引入第二套生产编辑器。
- **产品表达**：L3.0 仅是技术原型，不在普通用户界面显示“实时协作已支持”的承诺。

## 7. 阶段门禁

L3.0 只有在本 PRD、架构方案和 ADR 评审通过后进入 implementation。实现完成后进行一次集中 code-review + test-governance，再运行唯一 prototype aggregate。L3.0 全部验收通过才允许编写 L3.1 生产候选方案；失败则停止，不通过 fallback 或缩小断言范围宣称通过。
