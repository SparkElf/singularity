# 奇点 L2 异步协作产品需求

> L1 已完成企业空间、分享、审计、备份和显式内容身份。L2 在此基础上补齐异步协作，不承诺多人同时编辑同一文档。

## 1. 目标与范围

L2 让企业成员围绕一个明确文档进行可追踪的异步协作：提出问题、@相关人员、接收站内通知、查看和恢复历史版本，并按文档限制访问范围。正文仍由 Go Kernel 保持唯一事实源；评论、权限和通知属于企业控制面元数据。

### 1.1 本期包含

- 文档级评论线程：文档锚点、块锚点、回复、编辑、解决、重新打开和删除。
- `@` 提及：只可选择当前文档可见的组织成员或用户组成员，并为实际收件人生成站内通知。
- 站内通知：提及、回复、评论解决、文档权限变更和版本恢复通知；支持未读计数、已读和全部已读。
- 文档版本历史：版本列表、版本详情、差异查看和“恢复为新版本”；旧版本不可变。
- 文档级权限：继承空间权限或切换为受限模式；受限模式下按用户或用户组授予 `viewer`、`commenter`、`editor`。
- 与 L1 分享、审计、空间角色和现有设计系统的联动。

### 1.2 本期不包含

- 实时多人编辑、在线光标、在线状态、CRDT、冲突合并或编辑器事务广播改造。
- 邮件、短信、企业微信等外部通知渠道。
- 文档级拒绝规则、复杂 ACL 优先级、临时授权链接和跨组织成员同步。
- 跨空间全文搜索、离线编辑和本地内容事实源。

## 3. 角色与用户故事

### 3.1 角色

- **文档阅读者**：可查看文档和被授予的评论线程，不能修改正文或文档权限。
- **评论者**：可查看文档、创建和回复评论、@可见成员，不能提交正文事务。
- **文档编辑者**：拥有评论者能力并可提交正文事务，但不能修改文档 ACL，除非同时拥有空间管理权限。
- **空间管理员**：管理空间内文档 ACL、评论治理和分享联动；始终保留空间管理和访问能力。
- **组织 owner/admin**：在组织范围内保留管理能力，用于处理空间或文档锁定后的恢复。

### 3.2 用户故事

- 作为评论者，我可以在当前文档或一个真实块上提出问题，回复同一线程，并在问题解决后重新打开它。
- 作为被提及成员，我能在站内通知中看到来源文档、线程和提及者，并可直接回到仍有权限的文档。
- 作为空间管理员，我能把文档从继承空间权限切换为受限模式，并为用户或用户组授予明确的阅读、评论或编辑能力。
- 作为编辑者，我能查看某个文档的历史版本和差异，并将选定版本恢复为新的当前版本，不覆盖历史记录。
- 作为无权用户，我看不到文档评论、版本、通知来源或 ACL 细节，也不能通过修改 URL、请求体或分享地址越权。
- 作为管理员，我能从审计中追溯评论、提及、ACL 变更、版本查看和恢复操作的操作者与结果。

## 4. 领域模型与交互

### 4.1 文档级权限

文档默认处于 `inherit` 模式，直接继承空间角色。切换到 `restricted` 后，空间普通成员不再自动获得文档访问，只有文档 ACL grant 中的用户或用户组获得能力；组织 owner/admin 和空间 admin 始终保留管理能力。L2 不提供 `deny` grant，避免 deny/allow 优先级成为第二套权限语言。

Grant 能力从低到高为 `viewer < commenter < editor`：

- `viewer`：读取正文、历史和已授权评论。
- `commenter`：包含 viewer，并可创建、回复、编辑自己的评论、解决自己创建的线程和@可见成员。
- `editor`：包含 commenter，并可提交正文事务；仍不能自行修改 ACL。

分享读取必须同时满足分享仍有效和文档当前允许匿名读取的合同；切换到受限模式或撤销相关分享后，公开地址立即失效，不形成 ACL 绕过。

### 4.2 评论线程

评论线程只有一个权威内容身份：`organizationId + spaceId + notebookId + documentId`。块锚点额外携带 Kernel 返回的 `anchorBlockId`；文档锚点不伪造块 ID。锚点块被删除后，线程保留为未定位状态，不按标题、文本或相邻块自动迁移。

线程状态为 `open`、`resolved`、`deleted`。回复属于线程并按创建时间稳定排序；删除只隐藏正文内容并保留审计所需的不可变事件，不删除历史操作线索。

### 4.3 提及与通知

提及选择器只返回当前文档可见的活跃组织成员和用户组成员；提交时服务端重新解析收件人，不信任浏览器传入的显示名称或用户组成员快照。通知是 PostgreSQL 中的持久化站内收件箱记录，同一事件对同一收件人只生成一条未读通知。L2 使用页面进入、显式刷新和有界轮询读取，不新增 WebSocket 通知通道。

### 4.4 版本历史

历史版本由 Kernel 内容历史提供，API 只投影当前文档允许访问的最小版本元数据和差异。恢复动作通过正式内容 Gateway 提交，服务端先验证文档级 `editor` 能力，再让 Kernel 以当前三 ID 执行恢复；恢复结果形成一个新的当前版本和审计事件，旧版本永不覆盖或删除。

## 5. 关键流程

### 5.1 创建评论与提及

1. 用户从当前文档或块锚点打开评论编辑面板。
2. 输入正文和提及候选；候选只显示当前授权可见成员。
3. 提交时服务端校验当前文档能力、锚点身份和提及收件人资格。
4. 评论、提及通知和审计事件在同一控制面事务中落盘；任何失败都不返回成功评论。
5. 收件人打开通知时重新检查当前文档权限，无权时进入统一不可见结果，不泄露标题、正文或路径。

### 5.2 受限文档

1. 空间管理员在文档权限面板查看当前继承状态。
2. 切换受限模式并选择用户或用户组，授予最小必要能力。
3. 服务端事务锁定组织、空间、文档 ACL 和相关 grant，提交 ACL 变更及审计。
4. 已失去权限的连接在下一次 HTTP/WS 访问被拒绝；L2 不依赖客户端缓存自行收敛权限。
5. 现有分享按当前 ACL 重新评估，失效结果可被公开读取立即观察到。

### 5.3 查看和恢复历史

1. 用户在文档历史面板选择一个版本，服务端按当前三 ID和 ACL返回最小版本摘要。
2. 差异查看只展示该用户有权读取的内容，不把完整历史复制到 React 全局状态。
3. 具备 editor 能力的用户显式点击恢复并确认。
4. Kernel 返回 canonical 当前版本；控制面写入恢复审计和通知，页面刷新后从当前目录/Session重新读取。

## 6. 验收标准

每条标准都以用户可观察结果或跨层业务合同判定；源码字符串、私有函数或 mock 不能单独作为通过证据。

| 编号 | 验收标准 | 主要风险 | 最低充分证据 |
| --- | --- | --- | --- |
| L2-ACL-01 | `inherit` 文档与空间角色一致；`restricted` 文档只对显式 grant 和管理员可见 | 权限扩大或错误继承 | API contract + PostgreSQL integration |
| L2-ACL-02 | viewer/commenter/editor 的正文、评论和 ACL 能力严格分离 | 评论者获得正文写入 | 真实 HTTP + browser integration |
| L2-ACL-03 | 用户组 grant 的成员变化立即影响文档有效权限，不读取旧快照 | stale group cache 越权 | PostgreSQL integration + API HTTP |
| L2-ACL-04 | ACL 变更后现有分享按当前规则立即失效或恢复为明确结果 | 分享绕过文档权限 | 真实分享 E2E |
| L2-COM-01 | 文档和块评论始终绑定真实三 ID；删除锚点不迁移线程到近似块 | 跨文档评论或错误重锚 | contract + Kernel/HTTP integration |
| L2-COM-02 | 评论、回复、解决、重开、删除状态可从刷新后的服务端数据恢复 | 只存浏览器状态 | API integration + component |
| L2-COM-03 | 失权用户不能读取评论正文、作者、标题或锚点细节 | 内容元数据泄露 | cross-space HTTP + browser integration |
| L2-MEN-01 | 提及候选和提交收件人均按当前文档权限重新计算 | 浏览器伪造收件人 | API contract + integration |
| L2-NOT-01 | 评论/提及/ACL/恢复事件生成持久化站内通知，同一收件人不会重复未读 | 丢失或重复通知 | PostgreSQL integration |
| L2-NOT-02 | 通知入口显示未读计数，点击后回到仍有权限的文档或统一不可见结果 | 迟到通知泄露文档 | component + browser integration |
| L2-HIS-01 | 有权用户能查看当前文档版本列表和差异，无权用户不能读取历史 | 历史绕过 ACL | HTTP + Kernel integration |
| L2-HIS-02 | 恢复只生成新当前版本，旧版本仍可查看且审计记录操作者和结果 | 不可逆覆盖 | real HTTP + E2E |
| L2-OBS-01 | 评论、ACL、历史查看/恢复和通知均产出稳定审计，保留完整异常堆栈且不记录正文/令牌 | 无法追责或敏感信息泄露 | API/Worker logger integration |
| L2-BOUND-01 | 所有内容请求显式携带组织、空间、笔记本、文档身份，不从 DOM、全局状态、首响应或 URL 推断 | 跨库查询、迟到覆盖 | contract + browser + static boundary |

## 7. 约束

- **安全**：服务端是唯一权限 owner；评论和通知正文按普通用户内容处理，不执行 HTML、脚本、SVG、XML 或未知附件。
- **数据**：PostgreSQL 只保存评论、ACL、通知和操作元数据；正文、块、历史内容继续由 Go Kernel 持有。
- **一致性**：评论变更与通知/审计在同一 PostgreSQL 事务内提交；失败不返回局部成功。
- **异步性**：L2 使用持久化收件箱和刷新/轮询，不宣称实时推送或实时协作。
- **性能**：评论列表、通知列表、ACL grant 和版本摘要都分页；不把完整正文、完整历史或全量通知传入全局状态。
- **兼容**：沿用现有 React 19、Vite 8、Tailwind 4、shadcn、Zustand/React Query、NestJS、Prisma 和 Go Kernel。
- **设计系统**：新页面只使用 [奇点设计系统](../../output/md/Singularity_Design_System_v1.0.0_2026-07-13.md) 中的语义 token、shadcn 组件和既有 variants；评论状态、通知未读、ACL 能力和历史差异必须有统一状态样式。
- **国际化**：用户可见文案必须走现有语言资源边界，不在 API、测试或组件中硬编码开发者术语。

## 8. 阶段门禁

L2 只有在 PRD、架构方案、ADR 和任务计划获审阅后进入 implementation。所有五项能力及其迁移、调用方、永久测试和文档属于同一个大阶段；实现期间不执行正式 runner。全部功能收口后进行一次集中 code-review，再按架构方案的统一矩阵进入 verification。

## 9. 当前验证状态

生产实现、迁移、组件测试和唯一 P5 真实链路已完成；代码复评与 test-governance 复评通过。`lint:s0-s3`、`typecheck:s0-s3`、`build:s0-s3`、固定 PostgreSQL/API/Worker/Web 测试、Kernel service-auth 和 P5 E2E（12/12）已通过。三视口 browser integration 聚合仍未闭合：本轮 runner 报告 19 passed、40 failed、64 skipped，失败集中在既有页面的 Playwright 元素稳定性超时，不能作为 L2 全阶段绿色证据。L2 在该聚合门禁闭合前保持 verification 状态。
