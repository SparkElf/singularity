# Protyle插件兼容产品需求

## Change Log

| Version | Date | Author | Changes |
| --- | --- | --- | --- |
| 1.0.0 | 2026-07-13 | Codex | 固定P1-B2插件行为、边界与验收标准 |

## 1. 目标与范围

P1-B2将Protyle对旧`App.plugins`的直接读取收敛为一个插件端口，同时保持思源3.7.1现有编辑器插件的可观察行为。用户继续获得插件提供的编辑器选项、工具栏、事件菜单、快捷键、斜杠命令和粘贴处理；插件作者不需要为奇点维护第二套同义字段或调用顺序。

本批不新增插件管理页面，不迁移顶栏、Dock、页签、全局快捷键、文件树命令或插件市场，也不改变插件安装、启停和授权策略。上述工作分别属于React插件Facade、企业权限与后续真实编辑器接入阶段。

## 2. 同类调研

### 2.1 思源3.7.1本地官方源码

本仓基线`41f2861c87575ff5ac4b50a0520b1a4fe55b4a70`是当前兼容目标和权威事实源：

- `Plugin`公开`protyleOptions`、`updateProtyleToolbar`、`eventBus`、`commands`和`protyleSlash`。
- Protyle按插件注册顺序合并选项和工具栏、广播编辑器事件、匹配首个快捷键、列出并执行斜杠项。
- 菜单事件让插件向同一个菜单容器追加项目；paste事件以`preventDefault + resolve`表达异步接管，并把前一插件结果传给后一插件。

优点是行为已经被现有插件实际依赖；缺点是Protyle直接读取完整插件对象，事件、菜单和paste的不同语义被混在一个`EventBus.emit`布尔返回值中。奇点保留行为，不保留这项耦合。

### 2.2 现有奇点公共合同

P0已经建立`ProtylePluginPort`与类型擦除后的浏览器包入口，证明React和Protyle可以只共享能力合同。当前端口只提供options、toolbar、通用emit和dispose，尚不足以表达快捷键、斜杠身份、菜单聚合与顺序paste。

### 2.3 外部核对限制

本轮尝试只读获取思源官方`plugin-sample`，但当前环境访问GitHub Raw时发生TLS连接失败。由于本仓包含同版本完整官方源码、公开类型和运行时实现，产品决策以本地基线为准，不引用未取得的外部内容，也不以第三方博客替代官方行为。

## 3. 用户故事

- 作为知识库用户，我启用的编辑器插件在奇点中仍能修改编辑器选项和工具栏，不因React工作台替换而失效。
- 作为知识库用户，我能继续使用插件菜单、编辑器快捷键和斜杠命令，且同一次操作不会触发多个互斥命令。
- 作为知识库用户，多个paste插件按安装顺序处理同一份数据，后一插件看到前一插件的有效结果。
- 作为插件作者，我可以继续在公开事件detail或paste结果中携带自有额外字段，宿主不会因未知字段拒绝插件或静默剥离字段。
- 作为平台维护者，我能从Protyle插件端口定位兼容问题，不需要在编辑器各处追踪旧`App.plugins`。

## 4. 交互与状态

### 4.1 编辑器创建与工具栏

编辑器创建时按插件注册顺序应用`protyleOptions`，后应用的插件保持现有覆盖优先级。工具栏在初次创建和刷新时均按相同顺序扩展；插件自定义工具项继续使用用户配置的自定义快捷键。

没有启用插件时，编辑器选项和工具栏与插件迁移前保持一致，不出现空占位、禁用项或兼容提示。

### 4.2 事件与菜单

生命周期、编辑区点击、代码语言和菜单事件按注册顺序广播给全部启用插件。对代码语言列表和菜单容器的同步修改对当前操作立即可见；菜单项目仍显示在原有插件分组及分隔位置，不迁入React工作台状态。

### 4.3 快捷键与斜杠命令

编辑器快捷键按插件顺序和插件内部命令顺序匹配。首个匹配命令执行后终止本次插件命令查找；没有匹配时继续编辑器原有键盘路径。

斜杠列表保持插件顺序和插件内部顺序。选择项目时使用插件唯一标识和项目唯一标识定位原回调；重复项目标识只在各自插件范围内唯一，不依赖显示文案猜测目标。

### 4.4 Paste流水线

普通剪贴板和本地文件paste使用同一顺序规则。每个插件收到当前有效载荷；未接管事件的插件立即放行，调用`preventDefault`的插件可以异步`resolve`结果，流水线等待该结果后再进入下一个插件。

插件返回的有效字段更新当前载荷，未返回字段保留前一阶段值。额外自有字段继续传给后续插件；不设置超时、fallback或备用paste路径来掩盖未完成的插件回调。

## 5. 验收标准

| ID | 可判定结果 | 失败风险与关键路径 | 最低充分证据 |
| --- | --- | --- | --- |
| P-B2-01 | 多插件选项按注册顺序生效，后插件保持既有覆盖优先级 | 插件配置丢失或顺序反转；编辑器创建 | unit + static |
| P-B2-02 | 工具栏初建与刷新结果一致，插件项使用用户自定义快捷键 | 工具项消失或快捷键回退；工具栏生命周期 | unit + static |
| P-B2-03 | 编辑器事件广播给全部插件，detail的同步修改对当前操作可见 | 代码语言、生命周期或菜单扩展失效；事件广播 | unit |
| P-B2-04 | 插件菜单保持原分组和分隔位置，零插件时不生成空插件菜单 | 菜单错位或空菜单；五个直接菜单入口 | unit + browser integration（P4） |
| P-B2-05 | 首个匹配编辑器命令只执行一次，未匹配时不截断原键盘路径 | 多命令误触或按键失效；编辑器快捷键 | unit |
| P-B2-06 | 斜杠项保持稳定顺序，并以插件标识和项目标识调用准确回调 | 同名项误调用；提示列表与执行 | unit |
| P-B2-07 | paste按插件顺序等待异步接管并传递上一阶段结果，普通与本地文件载荷均成立 | 数据覆盖、乱序或提前上传；paste流水线 | unit + browser integration（P4） |
| P-B2-08 | 插件额外字段不被拒绝，paste额外字段可传给后续插件 | 插件扩展数据丢失；跨插件载荷 | unit |
| P-B2-09 | Protyle不再直接读取`App.plugins`；剩余`protyle.app`只属于已记录的B4非插件边界 | 旧壳依赖回流；浏览器入口边界 | AST + typecheck |
| P-B2-10 | 无插件时编辑、工具栏、菜单、快捷键、斜杠和paste保持原核心路径 | 基础编辑器被插件层阻断；全路径 | unit + 后续browser integration |

## 6. 约束

- **兼容**：以思源3.7.1本仓行为为准，不建立旧新双插件合同。
- **安全**：插件仍是用户或管理员明确安装的受信代码；本批不扩大其React工作台、组织权限或Kernel地址访问面。
- **性能**：快捷键不复制插件对象；paste只处理当前小载荷，不复制正文树或完整编辑器状态。
- **错误**：不吞插件异常，不用超时、默认结果或备用路径伪造成功。
- **字段**：插件标识、项目标识、事件类型和paste载荷各有唯一语义，不保留同义字段。
- **i18n**：不新增用户可见文案；插件自带HTML、文案和语言字段按原行为处理。
- **测试**：永久测试进入`pnpm test`；AST边界进入`pnpm lint`。目标后端被替代的浏览器测试只能称integration，不能称真实E2E。
- **产物**：文档、fixture、日志与测试输出只保留合同、结果和必要理由，不包含正文、凭证、内部提示或隐藏推理。
