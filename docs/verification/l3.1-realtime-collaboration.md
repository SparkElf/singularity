## 1. 结论

L3.1 已完成本阶段生产发布认证。产品需求、架构方案、ADR、实现、代码评审和测试治理均已收口；唯一 `verify:l3-release-certification` aggregate 在固定 PostgreSQL、受控 Go Kernel、真实 Nest/API/WSS 和真实浏览器链路上通过，包含容量上限、撤权、重启恢复、加密 admission、开关关闭、日志和回滚证据。认证范围仍严格限定为单 API 副本、单空间 Kernel；功能开关默认关闭，只有按发布判定表显式开启的试点或灰度环境可以使用。

验证确认 Go Kernel 继续是正文、块、AV、历史和 canonical operation log 的唯一事实源；Nest/PostgreSQL 只保存控制面元数据、ACL、会话和审计投影。每条跨边界消息显式携带 `organizationId + spaceId + notebookId + documentId`，协作模块不从 DOM、全局状态或首个响应推断身份。

| 默认开放判定 | 受控开放 | 功能开关继续默认关闭；发布只允许单副本/单空间试点或灰度 |

## 2. 锁定决策

| 决策 | 验证结论 |
| --- | --- |
| 部署 | 单 API 副本、单空间 Kernel 通道；不引入 Redis/NATS 或跨副本广播 |
| 正文事实源 | Go Kernel 负责正文、块、AV、历史和 canonical operation log |
| 控制面 | PostgreSQL 仅保存 ACL、开关、会话元数据和审计，不保存正文或 operation payload |
| 加密库 | 默认关闭；仅 `restricted-encrypted` 且成员、文档 ACL、内容密钥和客户端版本同时满足时允许协作 |
| 传输 | 专用双向协作 WSS；既有 Protyle 推送 WS 仍只读 |
| 身份 | 所有 join、submit、resume、presence、确认、冲突、撤权和关闭消息携带四段身份 |

## 3. 正式命令与环境

唯一正式技术验证命令：

```bash
cd /root/projects/singularity/enterprise
pnpm verify:l3-production
```

唯一正式生产发布认证 aggregate：

```bash
cd /root/projects/singularity/enterprise
pnpm verify:l3-release-certification
```

认证报告：`enterprise/test-results/l3-release-certification/report.json`；回滚演练报告：`enterprise/test-results/l3-release-certification/rollback.json`。

固定测试数据库已按项目配置保留：

| 项目 | 值 |
| --- | --- |
| 容器 | `singularity-postgres-test` |
| 地址 | `127.0.0.1:55432` |
| 数据库 | `singularity_test` |
| 持久卷 | `singularity-postgres-test-data` |
| 镜像 | `docker-compose.test.yml` 中锁定的 digest |

Docker Compose 插件不可用时，使用等价的 `docker run` 启动上述固定容器；本轮没有创建临时数据库。容器按用户要求保留，供后续开发复用。

Go Kernel case 使用 `CGO_ENABLED=1` 和 build tags `fts5 sqlcipher`，确保与生产内容存储能力一致。

## 6. 失败根因与修复记录

集中验证期间出现两组主要失败，均已完成整批修复并重新通过 aggregate：

| 根因 | 影响 | 修复 |
| --- | --- | --- |
| Go 测试未启用 `CGO_ENABLED=1`、`fts5`、`sqlcipher` | Kernel 内容存储相关 case 无法按生产能力构建/运行 | `verify:l3-production` 补齐环境变量和 `-tags "fts5 sqlcipher"` |
| Nest `DiscoveryService` 未被模块直接装配，且 `AccessChangedPublisher` 反射类型受循环依赖污染 | bootstrap 失败，声明式 handler/撤权链路无法建立 | `KernelGatewayModule` 直接导入 `DiscoveryModule`；以 `ACCESS_CHANGED_COLLABORATION` token/interface 解耦 Gateway |
| OpenAPI inventory 未同步新增 feature GET/PATCH 路由 | API 合同审计误报路由缺失 | 补齐 feature 路由 inventory、认证操作和 mutation header 合同 |
| 浏览器事务 route fulfill 与健康断言存在异步竞态 | attribute-view 用例在请求已接管但诊断仍 pending 时误报健康失败 | 在业务请求断言后等待既有 `pendingRequests` 集合清空；未改变生产请求链 |

失败归因均对应真实生产链路；未通过新增下游拦截来掩盖上游已经拒绝的不可达非法值。

## 7. 测试治理与边界审查

- **测试层级**：Contracts 使用标准 contract runner；Kernel 使用 Go 标准测试；API 和 Web 使用项目锁定的 unit/integration runner；静态规则使用结构化边界审计。
- **独立性**：case 同步注册，setup/cleanup 使用 runner hook；可按 case 过滤；没有兄弟 case 共享状态、逐文件子进程或顶层 `assert` 包装。
- **真实边界**：API/WS 使用真实 Nest bootstrap、固定 PostgreSQL 和受控 Kernel；Web 只在外部 Kernel 明确替换的 component/integration 边界使用替身，不宣称为真实 E2E。
- **声明式装配**：验证了 `DiscoveryModule`、DI provider、operation handler metadata 的真实发现、执行和冲突失败语义。
- **边界 owner**：协议 parser 首次解析四段身份；ACL 由 L2 `DocumentAccessPolicy` 唯一负责；Kernel 负责语义转换、加密 admission 和 history；下游不重复 parse、normalize 或重做权限算法。
- **重复数据流**：PostgreSQL 没有正文/operation payload 表；Zustand 没有正文快照；没有第二 CRDT 事实源、薄 adapter 或 fallback。
- **中文备注**：新增/修改的 bridge、join/submit、撤权关闭、加密 admission、resume 等关键函数均包含必要中文备注，说明作用、状态边界和副作用。
- **异常日志**：catch、重抛、WS error、Promise rejection 和 Kernel 转换保留原始 `name/message/stack` 及 request/session 关联；日志脱敏不包含正文、token 或密钥。

## 8. 发布和遗留风险

认证通过不等于无条件默认开放。当前仍需遵守：

1. 功能开关默认关闭，先进行内部试用和灰度容量观测。
2. 单副本达到容量或需要跨实例协作时，另立多副本/消息总线 ADR，不在本阶段增加兼容分支。
3. 加密库若无法在 Kernel 授权内容会话内取得密钥，必须返回 `encrypted-collaboration-unavailable`，不得降级为明文或旧快照。
4. API 重启会清除 presence，客户端须从 Kernel canonical history 缺口重新加入。
5. 认证只覆盖单 API 副本、单空间 Kernel；多副本、跨区域和灾备仍需另立方案和认证。
